# ✍️ Whiteboard Coding

> *"No autocomplete. No compiler. Just you, the marker, and your fundamentals."*

---

## I. THE REALITY
Many onsites (and some virtuals) use a whiteboard or a plain doc — no autocomplete, no syntax highlighting, no run button.

This requires KNOWING your language's API cold and planning before writing.

---

## II. THE LAYOUT
```
Top-left:    Assumptions + examples
Top-right:   Complexity
Center:      Main code (write LARGE, leave space between lines)
Bottom:      Trace / edge cases
```

---

## III. THE PROTOCOL
1. **Plan first** — pseudocode before real code (hard to insert lines later)
2. **Write large** with blank lines for corrections
3. **Use clear, short variable names**
4. **Trace by hand** to test (no compiler)
5. **Cross out**, don't frantically erase

---

## IV. MEMORIZE YOUR API
Without autocomplete, know by heart:
- C++: `sort`, `lower_bound`, `priority_queue`, `unordered_map`, iterators, `__builtin_popcount`
- Python: `sorted`, `heapq`, `collections.defaultdict/Counter/deque`, `bisect`
- Java: `Arrays.sort`, `PriorityQueue`, `HashMap`, `Collections`

---

## V. PRACTICE
- 1 problem/day on paper or whiteboard, no IDE
- Write full syntactically-correct code
- Then type into compiler — count syntax errors
- Drive errors toward zero

---

## VI. WHY IT'S WORTH IT
Whiteboard fluency makes IDE coding trivial. If you can code clean on paper, the IDE feels like a luxury.
